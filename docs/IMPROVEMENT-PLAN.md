# QuickButton — план улучшений

Документ собран по состоянию MVP. Цель — зафиксировать, что менять, зачем, в каком порядке и каким критерием считать, что задача закрыта.

Приоритеты:

- **P0** — блокеры качества, делаем в первую очередь.
- **P1** — сильный вклад в надёжность/UX, ближайшие итерации.
- **P2** — желательно, но можно отложить.

---

## Часть 1. Архитектура

### A1. Убрать дублирование main / preload (P0)

**Проблема.** Живут две параллельные реализации: `electron/main.cjs` + `electron/preload.cjs` и `src/main/electron.ts` + `src/preload/index.ts`. Каждый новый IPC-канал приходится править в 4+ местах. Уже приводило к рассинхронам.

**Решение.** Выбрать один источник правды.

Вариант A (быстрый): оставить только `electron/*.cjs`, удалить `src/main/*` и `src/preload/*`. Типы, которыми пользуется renderer, — в `src/shared/types.ts`.

Вариант B (правильный): собирать TS в CJS через `esbuild` / `tsup`, писать main и preload на TS, в `package.json.main` указывать `dist/main.cjs`. Ожидаемый бандл — <200KB.

**Критерий готовности.** Добавление нового IPC-канала требует правки ≤ 2 файлов: одного шаред-файла со схемой и одного обработчика.

### A2. Декомпозиция renderer (P0)

**Проблема.** `src/renderer/index.html` ~2300 строк: HTML, CSS, state, IPC, DnD, render, polling курсора.

**Решение** — без React/Svelte, только модули ES:

```
src/renderer/
  index.html          // только разметка + <script type="module" src="./main.js">
  styles/
    base.css
    editor.css
    grid.css
    service.css
  modules/
    state.js          // объект state + markDirty, loadPreset, savePreset
    render.js         // render(), render-панели
    dnd.js            // drag&drop кнопок/service/команд
    clickThrough.js   // polling курсора + setIgnoreMouseEvents
    ipc.js            // тонкая обёртка над window.quickButtonApi
    icons.js          // filePathToSrc, pickIcon
    validate.js       // sanitizePreset, validateCommand
  main.js             // init() + подписка на меню
```

**Критерий готовности.** Ни один модуль не превышает ~300 строк. `render.js` не импортирует IPC напрямую — только через `ipc.js`.

### A3. Единый store с подпиской (P1)

**Проблема.** State разбросан: `state.preset` мутируется в 30+ местах, плюс модульные переменные `selectedButtonId`, `draggingButtonId`, `clickThrough*`, `cursorPoll*`. Каждая правка требует ручного `render()`.

**Решение.** Минимальный store (50 строк):

```js
const store = {
  state: initialState,
  listeners: new Set(),
  update(mutator) {
    mutator(this.state);
    this.listeners.forEach((l) => l(this.state));
  },
  subscribe(l) {
    this.listeners.add(l);
    return () => this.listeners.delete(l);
  }
};
```

`render()` подписывается один раз, при вызове `update` — перерисовка. UI-состояние (`selectedButtonId`, режим DnD) живёт в `store.state.ui`.

**Критерий готовности.** Вызов `render()` пропадает из прикладного кода; остаётся только в подписке store.

### A4. Версионирование пресета + миграции (P0)

**Проблема.** Расширяем `ButtonStyle`/`UiConfig` без контроля версии. Старые пресеты могут молча терять данные.

**Решение.**

```ts
// src/shared/types.ts
export const PRESET_SCHEMA_VERSION = 2;
export interface Preset {
  version: number;
  meta: { createdAt: string; updatedAt: string };
  ui: UiConfig;
  grid: GridConfig;
  buttons: ButtonConfig[];
  contacts: Contact[];
}
```

```ts
// src/shared/migrations.ts
const migrations: Record<number, (raw: any) => any> = {
  1: (raw) => ({ ...raw, ui: { ...raw.ui, clickThroughBackground: true } }),
  2: (raw) => ({
    ...raw,
    buttons: raw.buttons.map((b) => ({ ...b, style: { wrapLabel: false, ...b.style } }))
  })
};
export function migratePreset(raw: any): Preset {
  /* пройти цепочку до PRESET_SCHEMA_VERSION */
}
```

**Критерий готовности.** При открытии пресета v1 из disk → автоматически мигрирует и сохраняет v2. Покрыто тестом.

### A5. Валидация IPC через схемы (P1)

**Проблема.** Все `ipcMain.handle` делают ручной `Boolean(payload?.x)`; нет единого места для валидации.

**Решение.** Добавить `zod` (~12KB) или минимальный ручной валидатор. Ввести помощник:

```ts
defineIpc(channel, schema, (payload) => {
  /* handler */
});
```

Каналы, payload-ы и ответы — в едином файле `src/shared/ipc.ts`, подключается и main-ом, и renderer-ом (через preload).

**Критерий готовности.** Ни один `ipcMain.handle` не читает поля из `payload` без предварительной валидации.

### A6. Сетевой слой: таймауты, reconnect, очередь (P1)

**Проблема.** TCP/UDP без политики переподключения и таймаутов. `continue on error` может оставить висящие сокеты.

**Решение.**

- Обёртки `UdpSender`, `TcpSender`, `OscSender` с контрактом:
  - `connect(timeoutMs)` — бросает `TimeoutError`.
  - `send(payload, { timeoutMs })`.
  - `close()` освобождает ресурсы.
  - единая очередь отправки per-контакт, чтобы не гонять несколько `send` параллельно по одному сокету.
- Структурированные ошибки: `{ code: 'ETIMEDOUT' | 'ECONNREFUSED' | 'EENCODE' | ..., message }`.

**Критерий готовности.** Smoke-тест на «выключенный сервер»: отправка UDP → ок; TCP → таймаут 2s, не подвешивает UI; OSC — то же.

### A7. Логи и диагностика (P1)

**Проблема.** Ошибки в полевых условиях не видны, кроме тостов.

**Решение.**

- `electron-log` (~40KB) или свой минимальный логгер: `log.info/warn/error` пишет в `userData/quickbutton.log` с ротацией.
- Меню `Help → Open logs folder`.
- Логировать: старт приложения, версия, загруженный пресет, каждый отправленный command (уровень `info`), ошибки сокетов (уровень `error`).

**Критерий готовности.** После сбоя у пользователя можно попросить прислать файл логов.

### A8. Тесты (P1)

**Проблема.** Есть только `m3-smoke.cjs`. Риск регрессий в OSC/TCP/валидации.

**Решение.**

- `vitest` + JSDOM.
- Покрыть:
  - `sanitizePreset`, `migratePreset`, `validateCommand` — unit.
  - OSC-кодер: известные входы → известные байты (фикстуры).
  - TCP framing: склейка пакетов по разделителю.
- CI-скрипт `npm test` должен идти ≤10s.

**Критерий готовности.** ≥20 тестов, зелёные; покрытие критичных модулей ≥80%.

### A9. Правильный polling курсора (P2)

**Проблема.** `setInterval(60ms)` гоняет IPC постоянно.

**Решение.**

- Пауза polling, когда `mainWindow.isFocused() === false` или `document.hidden === true`.
- Динамически увеличивать интервал до 250ms, когда курсор вне окна; возвращать к 60ms, когда внутри.
- Альтернатива: попробовать платформо-специфичные вызовы (Windows: `SetWindowLong WS_EX_LAYERED+WS_EX_TRANSPARENT` по регионам; macOS — NSWindow `ignoresMouseEvents` по hitTest).

**Критерий готовности.** Измерить CPU в Activity Monitor в `use` mode без активности: <0.5%.

### A10. Упаковка и подпись (P2)

- macOS: Apple Developer ID + notarize (`@electron/notarize`).
- Windows: code-sign (EV cert для моментального SmartScreen trust).
- Auto-updater: `electron-updater` + GitHub Releases как канал.
- README с инструкцией для разработчика: `npm run dist:mac -- --config.mac.identity=...`.

## Часть 2. Дизайн / UX

### D1. Визуальная иерархия редактора (P0)

**Проблема.** В редакторе кнопки в одной строке рядом лежат цвет фона, цвет текста, размер шрифта, перенос, радиус, выбор фона, удаление фона. Пользователь не видит групп.

**Решение.** Разбить на 3 секции с тонкими заголовками:

```
Label
  [input label]

Colors
  Background [color]   Text [color]

Text
  Size [n]   Wrap [toggle]

Shape
  Radius [n]

Background image
  [pick]  [clear]
```

У каждого контрола — микротекст снизу или слева.

**Критерий готовности.** В usability-тесте новый пользователь находит нужный контрол за ≤3 секунды.

### D2. Мини-preview выбранной кнопки (P0)

**Проблема.** Нет моментального ощущения, как кнопка будет выглядеть. Надо смотреть в грид, который может быть перекрыт.

**Решение.** В редакторе — карточка 120×72 с превью, применяются все текущие стили в реальном времени. Обновляется при любом изменении.

**Критерий готовности.** Изменение цвета/радиуса/переноса видно мгновенно в preview без взгляда на грид.

### D3. Читаемость текста поверх фоновой иконки (P1)

**Проблема.** На ярком PNG лейбл не виден.

**Решение.**

- По умолчанию `.user-btn.has-bg-icon` получает `text-shadow: 0 1px 2px rgba(0,0,0,.65)` + на низ кнопки тонкий градиент `linear-gradient(to top, rgba(0,0,0,.35), transparent 40%)`.
- В редакторе — slider `Background darken 0–100%`.
- Toggle `Label visibility: always / hover / never` (для чисто-иконочных кнопок).

**Критерий готовности.** Белый/чёрный текст на любом выбранном изображении остаётся читабельным.

### D4. Обратная связь по отправке команды (P0)

**Проблема.** `click-pulse` анимация есть, но нет статуса «доставлено / ошибка».

**Решение.**

- При успехе — зелёная точка в углу кнопки на 600ms.
- При ошибке — красная точка + toast с кратким текстом (`ECONNREFUSED 127.0.0.1:7000`).
- Глобальная иконка-индикатор последней операции в service-ячейке.

**Критерий готовности.** По отклику кнопки в `use` mode всегда понятно: «отправил» / «не отправил».

### D5. Горячие клавиши (P1)

**Проблема.** Инструмент для операторов без клавиатурных шорткатов — половина скорости.

**Решение.**

- Встроенные глобальные:
  - `Cmd/Ctrl+E` — переключение edit/use.
  - `Cmd/Ctrl+S` — save.
  - `Cmd/Ctrl+Shift+S` — save as.
  - `Cmd/Ctrl+O` — open.
- В `use` mode `Tab`/`Shift+Tab` — фокус по кнопкам, `Space/Enter` — send.
- Настраиваемый per-button shortcut (поле `hotkey` в `ButtonConfig`, регистрация через `globalShortcut`).

**Критерий готовности.** Все типовые действия доступны без мыши.

### D6. Индикация режима (P1)

**Проблема.** Edit и use визуально почти не отличаются.

**Решение.**

- В `edit` mode — тонкая полоска сверху окна цвета акцента + подпись `EDIT` в service-ячейке.
- В `use` mode — без полоски; service-ячейка чуть меньше/прозрачнее.
- Анимация перехода 150ms.

**Критерий готовности.** По одному взгляду на окно понятно, в каком ты режиме.

### D7. Подсказки для неочевидных фич (P1)

**Проблема.** `Click-through background`, DnD, service cell — все нужны, но неочевидны.

**Решение.**

- Иконка `(i)` рядом с каждой неочевидной настройкой, tooltip с 1-2 предложениями.
- При первом запуске — лёгкий onboarding overlay (3 шага): «1/3 Перетащи кнопки», «2/3 В use-mode фон прозрачен», «3/3 Service-ячейка — для перемещения окна».
- Сохранять факт просмотра в `userData/onboarding.json`.

**Критерий готовности.** Новый пользователь без чтения README осваивает edit + use.

### D8. Группы / страницы кнопок (P1)

**Проблема.** На 20+ кнопках экран превращается в кашу. Прямой конкурентный бенчмарк (Stream Deck, Companion, TouchOSC) везде имеет страницы.

**Решение.**

- Модель: `preset.pages: Page[]`, каждая — свой grid + buttons.
- Переключение страниц: tabs в service-ячейке или горячие клавиши `Cmd+1..9`.
- Специальный тип команды `switchPage(pageId)` — чтобы на кнопку можно было навесить переход.

**Критерий готовности.** Можно создать 3 страницы по 12 кнопок, переключаться клавишей и кнопкой.

### D9. Кастомный color picker (P2)

**Проблема.** Нативный OS-picker на macOS — гигантское окно, ломает overlay-контекст.

**Решение.** Лёгкий popover-picker (`@simonwep/pickr` или `coloris`). Палитра recent colors + 8 пресетов.

**Критерий готовности.** Выбор цвета не уводит фокус из приложения.

### D10. Типы контента в кнопке (P2)

**Проблема.** Сейчас на кнопке всегда и лейбл, и (опционально) фон-изображение.

**Решение.** Радио-группа `Display: Text | Icon | Icon + Text`. Для `Icon only` лейбл не рендерится, но используется как tooltip и для поиска.

**Критерий готовности.** Чисто-иконочные кнопки (без текста) рисуются корректно.

### D11. Поиск и быстрая навигация (P2)

При 50+ кнопках оператору нужен `Cmd+K`-палитра: ввёл «volume», мгновенно нашёл и запустил команду. Это не заменяет сетку, а дополняет её.

## Как измерять прогресс

- **Архитектурные метрики:** строк в `index.html` ≤ 200; число мест, где мутируется `state.preset`, ≤ 1 (store); тесты зелёные; миграция v1→текущая проходит; CPU в idle use-mode < 0.5%.
- **UX-метрики:** usability-тест «создай 5 кнопок, привяжи к ним UDP-команды, переключись в use, разберись с click-through» без подсказок ≤ 10 минут у нового пользователя.
